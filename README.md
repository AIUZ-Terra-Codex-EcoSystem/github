## What this organization is

The `AIUZ-Terra-Codex-EcoSystem` organization is a structured GitHub contour for
the Terra / FMP ecosystem.

It is meant to keep distinct layers distinct:

- theory;
- practice;
- legal and governance donor files;
- organization-level community and profile surfaces.

It is not meant to collapse every archive, backup, and mixed drafting layer into
one repository story.

## Layer logic

The organization should keep these layers distinct:

- theory repositories;
- practice repositories;
- publication repositories and DOI-facing sublayers;
- legal and donor repositories;
- design and philosophy bridges where they belong;
- archive and machine layers without letting them dominate the public root;
- organization profile surfaces;
- template and standardization mechanisms.

The profile should help people move across these layers without collapsing them
into one undifferentiated story.

## Template rule

The organization should expose one reusable template logic for new repositories.

That template logic should inherit:

- governance from `terra-legal`;
- publication and citation discipline from the shared bibliography layer;
- release and DOI workflow spine from the standard publication contour;
- layer routing from `GITHUB_LAYER_MODEL.md`;
- English-first human entry rules from the profile surface.

## Reading path

If you are entering the organization as a human reader:

1. start with this `README.md`
2. open [`profile/README.md`](./profile/README.md) for the profile-facing version
3. then read `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md`, `SECURITY.md`
4. then move into the specific repository layer you actually need
5. if you are creating a new repository, follow the donor and template rules instead of rebuilding the surface by hand

## Language policy

The public root surface should remain English-first.

Other languages may exist as secondary companion layers, but the main
organization entry should stay readable in English.

## Core principle

The organization should reduce entropy, not multiply it.

That means:

- no fake canonical files;
- no blind duplication across repositories;
- no machine-facing files pretending to be first human entry points;
- no unnecessary language confusion at the root layer.
